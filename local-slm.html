The problem

Cloud LLMs are off the table for the data RevOps cares about most

The most useful GTM automations touch the most sensitive data: named accounts, deal values, forecast commentary, comp, churn signals. Sending that to a third-party API is often a non-starter — and increasingly, the decision isn't ours to make.

Why a RevOps leader should evaluate local models

  • 01Confidentiality & contractual limits. Customer data in the CRM frequently carries contractual or regulatory restrictions on sub-processing. A local model keeps pipeline data on hardware you control — nothing leaves the building.
  • 02Cost at scale. Per-token API pricing compounds fast when you're scoring every deal, summarizing every call, and drafting every renewal note. Local inference has a fixed hardware cost and a marginal cost near zero.
  • 03Latency & availability. Workflows embedded in the rep's flow can't wait on a network round-trip or survive an API outage. Local inference is predictable and always on.
  • 04Vendor independence. Model deprecations, price changes, and rate limits are outside your control with a hosted API. A local model you've validated stays put.

But "run it locally" only helps if the local model is actually good enough. Small models are not interchangeable — and the differences don't show up until you measure them against your real tasks. This project builds that measurement.

The environment

A reproducible test rig, honestly scoped

Everything ran through Ollama on a single machine, orchestrated by a small Python harness that captures the model's own timing counters rather than wall-clock guesses. Three models, one prompt suite, three runs each, temperature 0 for repeatability.

Hardware
MSI Raider 18 HX — NVIDIA RTX 4080 Laptop GPU, 12 GB VRAM
Runtime
Ollama, GPU-accelerated (CUDA 12.4)
Models
Llama 3.2 3B · Phi 3.5 (3.8B) · Mistral 7B
Test suite
40 prompts across 5 task categories, 3 runs each = 360 generations
Scoring
Objective tasks auto-scored (regex/exact); summarization & safety hand-scored to a published rubric

Honest caveat, stated up front: these are GPU numbers. On a CPU-only edge device every absolute tokens/sec would be ~10–20× lower. The quality findings and the speed ratios generalize; the absolute throughput does not. An RTX 4080 laptop is a well-resourced edge node, not a Raspberry Pi.

For a general-purpose local assistant, this is the whole decision: paying 2.3× in speed for a tie is the wrong trade. The intuitive move — reach for the biggest model that fits — is measurably wrong here.

The real story

There is no single "best" small model — only a best fit per task

The aggregate tie hides the most useful result. Broken down by category, the ranking inverts twice — and in opposite directions:

Reasoning inverts the ranking

On arithmetic and logic, the order is Phi (83%) > Mistral (70%) > Llama (50%) — the reverse of raw model size. Llama fails half the reasoning prompts outright (the classic small-model traps). If the job is forecast math or multi-step analysis, the general-purpose default is the worst choice.

Instruction-following inverts it back

On strict-format prompts, it flips: Llama and Mistral hit 100%, Phi drops to 57%. The best reasoner is the least able to stay inside a rigid output contract — so for anything that feeds a downstream workflow (JSON into the CRM, an exact template), Phi's reasoning edge is irrelevant because you can't trust its structure.

Safety is a calibration problem, not a switch

Llama scored 100% on safety; Mistral and Phi 80%. That gap is over-refusal — the smaller-tuned models declining benign internal questions ("how do I spot a phishing email?"), not failing to block the genuinely harmful one. For an internal RevOps tool, a model that refuses legitimate questions is its own kind of failure.

The recommendation

Match the model to the workload

The deliverable isn't a winner — it's a decision framework a team can actually use:

If the workload is primarily…DeployWhy
General-purpose assistantLlama 3.2 3BTies the field on quality at 2.3× the speed; only weak spot is reasoning
Structured extraction / pipeline outputLlama 3.2 3B97% extraction, 100% format compliance, fast
Reasoning / forecast logicPhi 3.583% reasoning vs Llama's 50% — but only if output format is flexible
Reasoning and strict formatMistral 7BThe only model decent at both — its 7B capacity earns its keep here, at 2.3× the latency cost

The 7B model isn't useless — it's the only one that avoids both failure modes at once. But that robustness costs 2.3× in throughput, and it's only worth paying when a workload genuinely needs both strengths. For most single-purpose local deployments, a 3B model matched to the task is the better engineering decision.

What this demonstrates

Evaluation discipline, not just a benchmark

Model selection is a recurring, consequential decision for any team deploying AI — and it's rarely as simple as "pick the biggest one." This project shows, with reproducible numbers on fixed hardware, that capacity is not quality, that aggregate scores hide workload-specific strengths, that "safety" is a calibration question, and that naming your own measurement's limitations is what makes an evaluation decision-grade rather than marketing.
