Dozens of workflows, one architecture.

Each one is the same shape: a trigger, a retrieval, a reasoning step, a structured result, and a human at the gate. That's why a new workflow is a configuration, not a rebuild.

The last step is the one I never skip. The system does the gathering and the first draft; the judgment stays with a person wherever revenue, a customer, or someone's comp is on the line.

Intelligent Forecast
An executive-facing forecast dashboard for a B2B SaaS pipeline. It reads a bottom-of-funnel pipeline review export, applies deterministic rules grounded in MEDDPICC and deal hygiene, and does the two things a forecast call needs: flags which committed deals are at risk and why, and projects the number for a chosen timeframe (booked + open pipeline → a risk-adjusted forecast). The rules own every flag; the model only explains and routes so every number is auditable.

The hard problem

Three executives look at the same forecast and ask different questions: the CFO wants a number defensible to the board, the CRO wants where to point the team, the RevOps VP wants who owns the fix. A black-box model answers none of them credibly. The hard part is making a forecast that's simultaneously prioritized, projectable, and auditable down to the individual rule that fired.

Approach

A fully-tested deterministic rule engine (each rule maps 1:1 to a MEDDPICC or hygiene signal) scores a 600-deal pipeline; a FastAPI service + React SPA deploy as one container; an MCP tool server exposes the detector so agents can query it conversationally. The LLM only narrates and routes questions to one of four agents - it never sets a flag.

Key result
The detector scores itself against labeled ground truth. Region-aware scoring lifts it from F1 0.847 → 0.960 (precision 0.74 → 0.92, recall 1.00). Judging each deal against its own region's norm, not one global average, and recovers the +11.3 F1 points: it stops over-flagging EMEA's normally-long proposals and catches NA's fast-region stalls the global norm misses.
Stack
Python · React + Vite · FastAPI · pandas · Model Context Protocol (MCP) · Anthropic SDK · Docker · pytest / ruff / black · deterministic rule engine + multi-agent layer
Engineering rigor
  • Deterministic core makes zero network calls and runs end-to-end with no API key. The LLM paths degrade gracefully when unset
  • Real evaluation harness: precision / recall / F1 + confusion vs. labeled ground truth, reported per-rule, regenerable with one command
  • Every flag traces to one auditable rule built only from the deal's own values; a manager can verify it against the CRM
  • Period-over-period bookings math (YoY / QoQ / MoM / YTD) and a risk-adjusted projection by timeframe (stage win-rates, haircut on flagged deals, uplift on fast movers)
  • MCP server exposes the detector as read-only tools - structured JSON back, zero LLM calls in any tool
  • All thresholds live in config so a RevOps admin can retune without touching rule logic; all data synthetic

Deal Desk Helper
A citation-grounded contract-analysis tool that answers two questions a Deal Desk actually asks: "what does this contract say about X?" and "which contracts across our book have clause X?" It bridges the gap between how a business user phrases a question and how contracts are actually written.

The hard problem

Contracts rarely use the query's words. "Termination for convenience" appears as "terminate without cause," "in sole discretion," "without penalty." Keyword search misses them; even a cross-encoder reranker scores them near zero. An LLM judge recognizes the legal equivalence and recovers matches pure retrieval scored at zero.

Approach

Hybrid retrieval (BM25 + vector, RRF fusion) → cross-encoder rerank → citation-enforced generation, with an LLM-judge membership step for cross-corpus queries and query-rewriting that turns Deal Desk shorthand into contract language. Human-in-the-loop: it cites sources or declines rather than guessing.

Stack
Python · LangGraph · ChromaDB · Cohere (embed + rerank) · Anthropic Claude · Streamlit · custom LLM-as-judge eval + set-based precision/recall · Langfuse tracing · CI faithfulness + latency/coverage gates
Engineering rigor
  • Evaluation grounded in CUAD's expert annotations (100 real contracts, CC BY 4.0)
  • Faithfulness measured (claim-level), not asserted; declines handled correctly
  • Every stage traced (Langfuse): retrieved chunks, rerank reordering, exact prompt/response, tokens + cost per call
  • SRE metrics from request traffic — p50 21s / p95 60s (single-doc vs. analytical fan-out), 100% citation coverage, 0% failure rate
  • CI fails the build if faithfulness regresses, citation coverage drops below 80%, or p95 latency exceeds 90s
  • Diagnosed a reranker result-cap, a semantic gap, and term polysemy — each with data
  • Zero proprietary data; licensing + provenance handled as a first-class concern

Local SLM Evaluation for RevOps
A reproducible benchmark answering a question every RevOps leader now faces: when confidential CRM and pipeline data can't leave the building, which local model is actually good enough to run against it? Three small language models, on-premise, measured on the tasks a GTM assistant actually performs.

The hard problem

The most useful GTM automations touch the most sensitive data: named accounts, deal values, comp, and churn signals which often can't be sent to a third-party API. Running locally solves confidentiality, cost, and latency at once, but only if the local model is good enough. And small models are not interchangeable: the differences stay hidden until you measure them against your real tasks.

Approach

A Python harness runs 40 prompts across 5 RevOps-relevant categories (extraction, summarization, reasoning, instruction-following, safety) against each model, 3 runs each, capturing the model's own timing counters. Objective tasks auto-scored via regex/exact match; summarization and safety hand-scored to a published rubric, then normalized so categories compare fairly.

Key finding
The 7B model tied the 3B on overall quality (87.5% vs 87.6%) while running 2.3× slower - capacity ≠ quality. And the ranking inverts by task: the best reasoner (Phi, 83%) is the worst at strict formatting (57%). The deliverable isn't a winner. It's a workload-to-model decision framework.
Stack
Python · Ollama · Pydantic (constrained generation + validation + retry) · custom benchmark + scoring + aggregation harness · rubric-normalized quality scoring · GPU-instrumented (RTX 4080, CUDA 12.4)
Engineering rigor
  • Timing taken from the model's native counters - generation speed isolated from load cost
  • Median-of-runs methodology; determinism verified (temp 0 gave byte-identical output across 10 runs)
  • Two-guard reliability pattern: schema-constrained generation + Pydantic validation + single retry, failing gracefully
  • Honest limitations stated up front. GPU not CPU, so speed ratios generalize but absolutes don't
  • Safety scored as calibration (over- vs under-refusal), surfacing small-model over-refusal on benign prompts
